# 拼豆 App 设计开发上线推广计划

## 1. 当前团队配置

当前只有两个人：

- 全栈开发：负责 Flutter 客户端、算法实现、后端/数据能力、打包上线、技术验证。
- 产品/设计/推广：负责需求定义、用户流程、UI 设计、内容运营、用户反馈收集、上线推广。

因此第一阶段不适合追求“大而全”的产品，而应该围绕一个最短闭环推进：

`上传照片 → 裁切 → 选择尺寸和颜色 → 生成拼豆图纸 → 查看用料清单 → 保存/分享`

第一版的目标不是做完整商业平台，而是验证三个问题：

1. 用户是否愿意把照片转成拼豆图纸。
2. 生图效果是否达到“愿意保存/分享/下单”的水平。
3. 用户更关心哪类场景：头像、情侣照、宠物、二次元、礼物、亲子、手作教程。

## 2. 总体阶段划分

## 3. 阶段 0：方向确认

### 目标

把产品从“一个拼豆工具”收敛成“一个具体人群在具体场景下愿意使用的工具”。

### 要做什么

产品/设计负责：

- 明确首版目标用户。
- 明确首版主场景。
- 整理竞品或相似产品。
- 确定首版不做什么。
- 确定成功标准。

开发负责：

- 评估 Flutter 项目是否能承载第一版。
- 评估图片生成算法的可行性。
- 评估本地生成与服务端生成的边界。
- 确认第一版是否需要登录、后端、支付。

### 建议首版定位

第一版可以先定义为：

“把用户上传的照片快速生成可制作的拼豆图纸，并提供用料清单。”

暂时不要把第一版定义成：

- 拼豆社区
- 拼豆电商
- AI 创作平台
- 手作课程平台
- 复杂编辑器

这些可以后置。

### 阶段成果

- 一页产品定位说明。
- MVP 功能范围。
- 非 MVP 功能列表。
- 目标用户画像。
- 首版成功指标。

### 推进门槛

满足以下条件再进入下一阶段：

- 两个人对第一版范围没有重大分歧。
- 已明确第一版只验证一个核心闭环。
- 已明确哪些功能坚决不做。
- 已确认技术上可以本地生成基础图纸。

## 4. 阶段 1：设计与算法验证

### 目标

确认用户流程可理解，生成效果可接受。

这一阶段最重要，不是 UI 画得多完整，而是确定“这个产品生成出来的东西有没有价值”。

### 要做什么

产品/设计负责：

- 完成主流程 UI 原型。
- 明确页面结构。
- 明确每一步用户需要做的选择。
- 准备 20-50 张测试图片。
- 记录不同图片类型的生成效果。

开发负责：

- 完成基础生成链路。
- 做算法参数对比。
- 支持不同尺寸、颜色数、平滑开关。
- 输出用料清单。
- 支持保存结果。

### 重点验证内容

需要重点验证这些参数：

- 图纸尺寸：`24x24`、`32x32`、`48x48`、`64x64`
- 颜色数量：`16`、`24`、`32`
- 平滑：开 / 关
- 抖动强度：低 / 中 / 高
- 图片类型：人像、宠物、二次元、logo、风景、情侣照

### 建议做一个内部算法对比页

同一张图一次生成多个版本：

- `16色 + 平滑关`
- `16色 + 平滑开`
- `24色 + 平滑开`
- `32色 + 平滑开`

这样可以快速判断默认参数。

### 阶段成果

- Figma 主流程设计稿。
- 可跑通的算法 Demo。
- 生图参数说明文档。
- 20-50 张样本图的生成结果。
- 推荐默认参数。

### 推进门槛

满足以下条件再进入 MVP 开发：

- 至少 60%-70% 测试图片能生成“可接受”的图纸。
- 人像、宠物、二次元至少有一个场景效果明显可用。
- 已确定默认生成参数。
- 产品流程不超过 4-5 个核心步骤。
- 不需要解释，用户大致能看懂每一步要做什么。

## 5. 阶段 2：MVP 开发

### 目标

做出可以安装、可以完整体验、可以给外部用户测试的第一版。

### MVP 功能范围

必须做：

- 上传照片。
- 图片裁切。
- 选择图纸尺寸。
- 选择颜色数量。
- 选择颜色品牌。
- 生成拼豆图纸。
- 查看用料清单。
- 保存结果。
- 基础编辑：替换颜色、删除颜色或简单涂改。

可以做但不强求：

- 分享图片。
- 导出 PDF。
- 历史记录。
- 简单教程页。

暂时不做：

- 登录注册。
- 社区。
- 支付。
- 商城。
- 多端同步。
- 复杂图层编辑。
- AI 文生图。

### 开发重点

开发负责：

- 清理主流程代码。
- 固化数据模型。
- 完成图纸生成服务。
- 完成本地存储。
- 完成错误处理。
- 完成基础性能优化。
- 完成 iOS/Android 打包。

产品/设计负责：

- 补齐缺失状态。
- 设计加载中、失败、空状态。
- 准备默认文案。
- 确认图标、颜色、字体和页面细节。
- 准备测试用例。

### 阶段成果

- 可安装测试包。
- MVP 功能清单。
- 已知问题清单。
- 测试图片样本库。
- 内测反馈表。

### 推进门槛

满足以下条件再进入内测：

- 主流程成功率稳定。
- 生成失败有明确提示。
- 普通图片生成时间可接受。
- 用料清单数量正确。
- 保存结果可用。
- 没有明显崩溃。
- 两个人可以连续完整跑 20 次流程。

## 6. 阶段 3：内测打磨

### 目标

让真实用户试用，找到体验问题和价值信号。

### 内测用户建议

第一批不需要太多，建议 10-20 人：

- 手作爱好者
- 拼豆玩家
- 宠物主人
- 情侣/头像用户
- 二次元头像用户
- 设计师或视觉敏感用户

### 要收集什么

产品/推广负责：

- 用户是否能独立完成流程。
- 用户在哪一步犹豫。
- 用户是否满意生成效果。
- 用户是否愿意保存或分享。
- 用户是否愿意真的照着做拼豆。
- 用户愿意为什么功能付费。

开发负责：

- 修复崩溃和阻塞问题。
- 优化生成速度。
- 优化内存占用。
- 优化保存和恢复。
- 根据反馈调整默认参数。

### 关键问题

内测最重要的不是问“你觉得好不好”，而是观察：

- 用户有没有主动换图再试一次。
- 用户有没有保存结果。
- 用户有没有发给别人看。
- 用户有没有问“在哪里买这些豆子”。
- 用户有没有想改颜色或手动编辑。

### 阶段成果

- 内测反馈报告。
- Top 10 问题列表。
- Top 5 用户需求。
- 生图效果优化清单。
- 第二轮测试版本。

### 推进门槛

满足以下条件再进入上线准备：

- 70% 以上内测用户能完成主流程。
- 至少 30% 用户愿意保存或分享结果。
- 生成效果的主要问题已经有解决方案。
- 没有明显崩溃和卡死。
- 已明确上线版本保留哪些功能、砍掉哪些功能。

## 7. 阶段 4：上线准备

### 目标

把产品从“能用”变成“可以公开发布”。

### 产品准备

产品/推广负责：

- App 名称。
- App 图标。
- 应用商店截图。
- 应用描述。
- 关键词。
- 隐私政策。
- 用户协议。
- 首批推广素材。
- 新手引导文案。

开发负责：

- App 打包配置。
- 版本号管理。
- 崩溃日志。
- 基础埋点。
- 隐私权限说明。
- 图片权限说明。
- 应用市场审核问题处理。

### 建议埋点

第一版至少记录：

- 上传照片点击。
- 裁切完成。
- 选择尺寸。
- 选择颜色数量。
- 生成开始。
- 生成成功。
- 生成失败。
- 保存结果。
- 分享结果。
- 进入编辑页。

这些数据能帮助判断用户卡在哪一步。

### 阶段成果

- 上线包。
- 应用商店素材。
- 隐私政策和用户协议。
- 埋点事件表。
- 首批推广内容。

### 推进门槛

满足以下条件再上线：

- 审核材料完整。
- 权限说明清楚。
- 关键流程没有阻塞 bug。
- 有基础数据追踪。
- 有上线后 7 天的推广计划。
- 有用户反馈入口。

## 8. 阶段 5：首轮推广

### 目标

获得第一批真实用户和真实反馈。

### 推广方向

优先尝试内容平台，而不是先投广告。

可以尝试：

- 小红书：拼豆、手作、情侣礼物、宠物周边。
- 抖音：照片变拼豆图纸的前后对比。
- B 站：拼豆教程、App 生成图纸过程。
- 微信群/豆瓣/贴吧：手作和拼豆社群。
- 闲鱼/淘宝店主合作：提供图纸生成工具或引流。

### 内容选题

第一批内容可以围绕：

- “把情侣照变成拼豆图纸”
- “把宠物照片做成拼豆挂件”
- “头像变拼豆教程”
- “一张照片生成拼豆用料清单”
- “不会画图也能做拼豆”

### 推广节奏

建议首轮 2 周：

第 1-3 天：

- 发布 3-5 条内容。
- 找身边用户试用。
- 收集第一批反馈。

第 4-7 天：

- 根据反馈改默认参数和文案。
- 发布对比类内容。
- 尝试进入手作社群。

第 8-14 天：

- 观察哪个内容方向有点击。
- 重点复制有效内容。
- 收集真实用户案例。

### 阶段成果

- 第一批真实用户。
- 第一批内容数据。
- 第一批用户案例。
- 用户反馈池。
- 下一版需求优先级。

### 推进门槛

满足以下条件说明值得继续投入：

- 用户会主动上传多张图片尝试。
- 用户会保存或分享生成结果。
- 有用户问材料包、购买、教程或定制。
- 有某一类内容明显比其他内容更容易传播。
- 反馈集中在“想要更好效果”，而不是“我不知道这个有什么用”。

## 9. 阶段 6：迭代增长

### 目标

从工具 Demo 变成可持续产品。

### 可能的第二版方向

根据首轮数据选择，不建议同时做。

方向一：图纸质量提升

- 人像优化。
- 背景弱化。
- 肤色保护。
- 效果模式。
- 更好的调色板。

方向二：编辑能力增强

- 手动画笔。
- 颜色替换。
- 局部重绘。
- 撤销恢复。
- 网格/编号显示。

方向三：导出与制作

- PDF 导出。
- 分区图纸。
- 编号图纸。
- 打印版。
- 教程步骤。

方向四：商业化

- 材料包清单。
- 购买链接。
- 定制图纸。
- 高级导出。
- 会员或单次付费。

方向五：内容和社区

- 模板库。
- 用户作品。
- 热门图纸。
- 作品分享。
- 活动挑战。

### 阶段成果

- 第二版产品路线图。
- 明确的核心场景。
- 明确的留存或转化指标。
- 明确商业化假设。

### 推进门槛

继续投入的判断：

- 有一个用户场景明显成立。
- 用户愿意保存、分享或实际制作。
- 生图质量问题可通过算法迭代解决。
- 推广内容能带来稳定自然流量。
- 有潜在商业路径。

## 10. 两个人的协作方式

### 每周节奏

建议每周固定一次 60-90 分钟同步：

- 看本周用户反馈。
- 看生成效果样本。
- 决定本周只做 1-3 个重点。
- 砍掉不重要的需求。
- 明确下周交付物。

### 日常协作

产品/设计每天可以给开发的输入：

- 一张明确设计图。
- 一个明确交互说明。
- 一个明确优先级。
- 一个真实用户反馈。

开发每天可以给产品/设计的输出：

- 一个可体验版本。
- 一个技术风险说明。
- 一个生成效果对比。
- 一个需要产品决策的问题。

### 决策原则

两人团队最怕“什么都想做”。建议遵守：

- 能用真实用户验证的优先。
- 能提升生图效果的优先。
- 能缩短主流程的优先。
- 能产生保存/分享行为的优先。
- 复杂但不影响验证的后置。

## 11. 第一版建议时间表

如果两人都能持续投入，可以按 6-8 周推进。

| 周期 | 重点 | 主要成果 |
| --- | --- | --- |
| 第 1 周 | 产品定位、需求收敛、算法验证 | MVP 范围、主流程、测试样本 |
| 第 2 周 | UI 设计、算法对比、默认参数 | Figma 主流程、参数结论 |
| 第 3-4 周 | MVP 开发 | 可安装测试包 |
| 第 5 周 | 内测和修复 | 内测反馈、优化版本 |
| 第 6 周 | 上线准备 | 商店素材、埋点、隐私政策 |
| 第 7-8 周 | 首轮推广和迭代 | 真实用户、内容数据、第二版方向 |

如果时间更紧，可以先做 4 周版本：

| 周期 | 重点 | 主要成果 |
| --- | --- | --- |
| 第 1 周 | 主流程和算法效果 | 可生成图纸 |
| 第 2 周 | MVP 开发 | 可安装包 |
| 第 3 周 | 内测修复 | 稳定版本 |
| 第 4 周 | 小范围发布 | 第一批真实反馈 |

## 12. 当前最应该做的事情

基于当前状态，下一步建议按这个顺序推进：

1. 完成算法对比页，确定默认参数。
2. 用 20-50 张真实图片测试生图效果。
3. 明确第一版主打场景：人像、宠物、二次元或礼物。
4. 完成主流程 UI 的最终版。
5. 开发可安装 MVP。
6. 找 10-20 个真实用户内测。
7. 根据反馈决定上线版本范围。

第一版的关键不是功能多，而是让用户第一次上传照片后能产生一句话：

“这个图纸我真的想保存下来，甚至想照着做。”
